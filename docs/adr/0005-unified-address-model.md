# Unified node-facing Address model, mapped per endpoint

The same address concept uses three different field-name conventions across Purolator's APIs: Estimate (`streetAddress` / `provinceStateCode` / `postalZipCode`), Pickup (`address` / `province` / `postalCode`), and Locator (`streetAddress` / `provinceCode` / `postalCode`). Rather than mirror each API's names, the node exposes **one** canonical Address structure with consistent field names (`street`, `city`, `province`, `country`, `postalCode`, optional `companyName`) and a thin per-endpoint transform maps it to each API's native names. Pickup's separate contact fields (`name`, `phoneNumber`) stay distinct from the address.

Chosen for cross-operation consistency (a node-quality signal reviewers look for, and a single mental model for users), a cleaner AI-tool schema (Principle 11 — the agent learns one address shape, not three), and because the mapping is exactly the payload-assembly transform Constitution Principle 10 already requires unit tests for. The trade-off: node field names diverge from Purolator's raw docs (documented in the README), and renaming these user-facing fields later breaks saved workflows — hence recording the contract here.
